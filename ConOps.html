The Current System

The current system design is an encrypted email system through a web browser based interface. Messages will be encrypted end-to-end so that they can only be read by the designated reciever. Unregistered users will be able to register a new account. Registered users can login, send/recieve messages and manage their messages.

 

The Proposed System: Needs

  • End to end encryption of messages
  • Organization of messages
  • Marking Spam
  • Finding unread messages
  • Interface to send, receive and manage messages
  • Server to handle sending/receiving of messages

 

The Proposed System: Users and Modes of Operation

The proposed system will have two types of users (admins and standard users). Standard users can send and receive messages. Admins can manage users and the site as well as sending/receiving messages. The proposed system will have two modes of operation (authenticated and non-authenticated). When in the authenticated mode, users can send/recieve messages as well as viewing and managing older messages. In non-authenticated mode, users can either register or login in (to switch to authenticated mode).

 

The Proposed System: Operational Scenarios

  • Create New User - An un-authenticated user registers for an account. Basic information is collected on them (desired username, pw, etc.) Basic password requirements check is caried out as well as checking to make sure the username doesn't already exist. If requirements are met, their account is created and they are logged in.
  • Login - A registered un-authenticated user logs into the website using their username and password. If they provide the correct information, they are authenticated and redirected to their inbox.
  • Send Message - An authenticated user wishes to send a message to another user. The message is encryped before it is sent.
  • Read Message - An authenticated user wishes to read a message recieved from another user. The message must be decrypted before it is readable by the user.
  • Logout - An authenticated user wishes to logout. Their active session is ended and they must login again to send/recieve messages.

 

The Proposed System: Operational Features

Must Have:

  • End to end encryption of messages
  • Ability to send messages to more than one user
  • Ability to send/recieve messages
  • Ability to manage messages (delete, sorting, etc.)
  • Browser based interface
  • Ability to create new user
  • Ability to authenticate registered users

Would Like to Have:

  • Sorting of messages based on various fields
  • Spam filter
  • Mobile based app (Android)
  • Password reset
  • Ability to search messages

 

The Proposed System: Expected Impacts

The expected impact is that users using our system will be able to communicate over email securely via end to end encryption.

 

The Proposed System: Analysis

Expected Improvements:

  • Implementation of must haves
  • Android App Interface

Disadvantages:

  • No Spam Filter
  • Encryption can be clunky and some users may have some dificulty using it

Limitations:

  • No way to attach files to messages
  • The current system only provides means to encrypt plain text, there is no way to embed formatting into messages

Risks:

  • If messages aren't properly enrypted, they could be intercepted and read by someone who isn't the intended reciever
  • The web server runs the risk of being hacked, which could lead to user data being leaked
