Project 0verview

The app will encrypt emails from one end to another so that someone intercepting the email will not be able to read it. It will allow for users to sign up, login, create, send, and recieve emails from other users on the system. The emails will be stored in a database and the server will send and recieve JSON. This will allow for a front-end web app and mobile app to be implemented.

Software Life Cycle Process

Our group will follow the Phase Development Process for this application. Each required part of the application will be worked on separately. Once each part is working, extra features will be implemented for each part. This process will allow us to develop and test each part of our application separately which will be easier to maintain in the long run.

Tools and Computing Environment

The operating system that we will try to target is android. That is a desirable feature that we will try to implement. The other platform will be on the web. We will be using Ruby on Rails for the server and AngularJS for the front end of the site. The mobile app will be written in Java.

Configuration Management

Our group will be using GitHub for version control. The project lead will be the one responsible for watching over. We will be developing on separate branches for each feature.

Quality Assurance

Each feature will be developed on a separate branch. Once the feature is ready to be merged the feature lead will send a pull request. The project lead will then look it over and either merge/decline the request. The features will be tracked on GitHub's issue tracker.

Risk Management

Software Part:

 

- Problem with Server: Some team members may not have a detailed understanding of JSON. We will find suitable material to research and minimize project delay

 

- Problem with GUIs: Should a group member fall behind then each team member will assist in bringing the group member back to par.

 

Communication:

 

- Miscommunication: Discussing everything over GroupMe may lead to a miscommunication of ideas or progress. If this were to become an issue, then we have to set up the time to meet after class.

 

- Scheduling: With the range of schedules between our group members, there may be conflicts when scheduling meetings. The way to solve this is to keep update the process on GitHub and GroupMe, so the group member can catch up by reading it.

 

Others:

 

- Team member withdraws from class: Then we should work harder and share his/her works.

 

- Time Management: If there are many assignments or exams in the same deadline of the project, project may not be able to follow the plan (or may not be done) at the deadline.

 

Table of Work Packages, Time Estimates, and Assignments

The work packages will be separated into networking, GUI, and security algorithms. It should take us two weeks for each part to be in a workable state. Thomas will be in charge of the network feature, Ryan will be in charge of the GUI, and David will be in charge of the security algorithms.

Technical Progress Metrics

The requirement phase will be measured by how many requirements have been implemented versus the number of required requirements that the application should have. The application design phase will be measured by the number of UML charts that have been contructed for the application. The development phase will be measured by the number of working features and sub features that have been implemented.

Plan for tracking, control, and reporting of progress

Every week, each team member will be developing the application. Work history will be trackable through git and resolved/opened issues will be tracked by GitHub. The project manager will review the work done for the week by each team member and submit a report based on how far development has progressed for the week. He will also communicate if any requirement or application features have changed
