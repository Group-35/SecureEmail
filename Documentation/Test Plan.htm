SECTION 1: Introduction

  • Overall Objective for Software Test Activity:
  • The software tests shall enable us to debug and catch potential errors in our program before deploying the application to a live environment. It will also allow us to refactor and optimize our code while ensuring that our application will still run as intended.

Reference Documents:

  • Concept of Operations

SECTION 2: Description of Test Environment

We will use MiniTests to test our server for any potential errors or security flaws. We will run integration and unit tests to ensure everything is working as intended. We will run tests on the server and the front end of our application. The testers will be the developers for the application. The testing environment will be different from the actual production environment. The main differences will be server configurations and database values.

SECTION 3: Stopping Criteria

When an error occurs while testing, we will stop testing and attempt to fix the problem. Once there are no more errors, we will deliver the application. If an error occurs during run time, then we will attempt to replicate the error in the testing environment and fix it. With the nature of the application, it should not take long to find all of the errors when running the testing suite. The developer running the tests will attempt to fix the bugs or put in a ticket for the error if they are not able to fix the problem. Before we develop a feature, we will make tests that it needs to pass. Then we will develop and fix those errors. Once it passed all of the errors then the feature will be ready. Things that are not application pending are cosmetic errors. Any other type of error is a 'fatal error' and we will attempt to fix it before delopyment.

SECTION 4: Description of Individual Test Cases

  • Test Objective: Email sent verified
    • Test Description: Test the ability to send emails from other users. The testing values will be the email object. This includes the to, from, and other email data.
    • Test Conditions: See Test Environment.
    • Expected Results: The expected result should be an additional value in the email database. The testing suite should also show "green" to notify that the case passed.
  • Test Objective: Email recieved verified
    • Test Description: Test the ability to recieve emails from other users. The testing values will be the email object. This includes the to, from, and other email data.
    • Test Conditions: See Test Environment.
    • Expected Results: The expected result should be that the email 'read' flagged was changed from false to true.
  • Test Objective: User login/logout
    • Test Description: The user should be able to login and logout of their session.
    • Test Conditions: See Test Environment.
    • Expected Results: The expected result should be the user loggin into the system and logging out by deleting their session. Another user should not be able to access another users data either.
  • Test Objective: Save drafts
    • Test Description: The user should be able to write and save a draft without sending the email.
    • Test Conditions: See Test Environment.
    • Expected Results: The expected result should be that the draft is saved to the server with no 'to' fields.
  • Test Objective: Secure sessions
    • Test Description: Another user should not be able to access another user's session by session hijacking.
    • Test Conditions: See Test Environment.
    • Expected Results: The expected result should be a user not being able to access data from another user.
